## v2.0.0 — Strategic bets (research required first)

### Bet 1: Replace vm sandbox with WebAssembly-based isolation

**Problem:** The current vm sandbox (`vm.runInContext`) is explicitly documented by Node.js as unsafe for untrusted code. Without Docker (which adds ~500MB and requires root), there's no safe code execution path.

**State of the art:**
- **Pyodide** (https://pyodide.org/) — CPython compiled to WebAssembly, runs Python in a WASM sandbox with no filesystem access. Used by Pyodide.org, JupyterLite, and CoCalc. MIT licensed.
- **wasmer-js** (https://github.com/wasmerio/wasmer-js) — Run WASI modules in the browser/Node.js with filesystem isolation. Used by Wasmer Edge for serverless functions.
- **Cloudflare Workers** (https://workers.cloudflare.com/) — Uses V8 isolates (not WASM) but demonstrates that sandboxed code execution at scale is possible without containers.

**Implementation approach:**
1. For Python: replace `execFileSync('python3', ...)` with Pyodide via `@pyodide/standalone`. Pyodide runs in a WASM sandbox with no `os` module access by default. Load time ~2s (one-time), execution ~1.5x native speed.
2. For JavaScript: replace `vm.runInContext` with `wasmer-js` running a QuickJS WASM build. QuickJS compiled to WASM gives true isolation (separate heap, no prototype chain access to host).
3. Both approaches work without Docker, root, or native dependencies. The WASM modules are ~10MB (Pyodide) and ~2MB (QuickJS-WASM).

**Risks:**
- Pyodide doesn't support all Python packages (no `numpy` in base, requires rebuilding). For research/analysis code (pure Python), this is fine.
- WASM execution is slower than native (1.5-3x). For the 10-second timeout use case, this is acceptable.
- Initial load time adds latency to first code execution.

**Validation criteria:**
- Run the 5 sandbox escape attempts from Phase 3.6 — all must be blocked
- Execute 10 Python code samples (statistical calculations, string processing) — all produce correct output
- Measure: cold start < 3s, warm execution < 1.5x native

### Bet 2: Add pgvector semantic search for production-grade memory recall

**Problem:** The current memory system uses SQLite LIKE search, which cannot find semantic matches. "I prefer Python" won't recall when the user asks about "my favorite programming language."

**State of the art:**
- **pgvector** (https://github.com/pgvector/pgvector) — Postgres extension for vector similarity search. 9.5K stars. Used by Supabase, Neon, Railway. Supports cosine, L2, and inner product distance. HNSW index for approximate search.
- **ChromaDB** (https://www.trychroma.com/) — Embedding database with built-in vector search. Open source. Simpler than pgvector but adds a separate service.
- **LanceDB** (https://lancedb.com/) — Embedded vector database (no server needed). Uses Arrow format. Good for single-user scenarios.

**Implementation approach:**
1. Enable pgvector in `prisma/schema.prisma` (uncomment `Unsupported("vector(1536)")`)
2. Add migration: `CREATE EXTENSION IF NOT EXISTS vector;`
3. In `memory-recall.ts`, use raw SQL `$queryRaw` with pgvector cosine distance operator `<=>`
4. Generate embeddings using NVIDIA nv-embed-v1 (free) or OpenAI text-embedding-3-small
5. SQLite path: keep LIKE search as documented fallback (no pgvector equivalent for SQLite)
6. Hybrid search: combine vector similarity (semantic) + LIKE (keyword) for best results

**Risks:**
- pgvector requires Postgres — increases deployment complexity
- Embedding generation adds latency (~100ms per memory extraction)
- Vector index tuning (HNSW vs IVFFlat) requires testing on real data volume

**Validation criteria:**
- Store 100 memories, recall accuracy > 80% for semantic queries
- Recall latency < 50ms with HNSW index
- SQLite fallback still works for development

### Bet 3: Add streaming agent swarm with real-time collaboration

**Problem:** The current swarm runs workers in parallel but the user sees nothing until synthesis completes. For a 3-minute research task, the user stares at a spinner for 3 minutes.

**State of the art:**
- **ChatDev** (https://github.com/OpenBMB/ChatDev) — Multi-agent framework where agents collaborate in a virtual software company. Shows real-time agent messages. 25K stars.
- **CrewAI** (https://github.com/joaomdmoura/crewai) — Agent orchestration framework with task delegation and real-time output. 19K stars. Uses a sequential or hierarchical process model.
- **Microsoft AutoGen** (https://github.com/microsoft/autogen) — Multi-agent conversation framework. Agents can talk to each other and the user simultaneously. 32K stars.

**Implementation approach:**
1. Extend the SSE event stream to include per-worker token streaming (already partially implemented in `swarm.ts`)
2. Add a "worker chat" phase: after initial parallel research, workers share findings with each other and can request clarification
3. The synthesizer sees the full conversation, not just final outputs
4. UI: show each worker's output in a separate panel (already prototyped in `SwarmCard.tsx`)
5. Add a "critic" worker that reviews other workers' outputs before synthesis

**Risks:**
- Worker-to-worker communication increases LLM calls (3-5x more tokens)
- Conflict resolution between workers is hard — the critic may not catch all disagreements
- Latency increases: from 3 min (parallel) to 5-7 min (collaborative)

**Validation criteria:**
- User sees worker output within 10 seconds of starting
- Final report quality (measured by eval harness) improves by >10% vs non-collaborative swarm
- Total time increase < 50% vs current parallel-only approach

### Bet 4: Add multi-tenant isolation with row-level security

**Problem:** The project is single-user by default. NextAuth v4 is integrated but there's no data isolation between users — any authenticated user can access any conversation, memory, or research job.

**State of the art:**
- **Postgres Row-Level Security (RLS)** — Built-in Postgres feature that filters rows based on the current user. Used by Supabase Auth, Neon, and most multi-tenant SaaS. No application code changes needed — the database enforces isolation.
- **Prisma multi-tenancy** — Prisma supports tenant isolation via `extends` with a tenant filter. Less secure than RLS (application-level, not database-level) but simpler.
- **Cabin** (https://cabin.dev/) — Multi-tenant Next.js template using RLS. Good reference implementation.

**Implementation approach:**
1. Add `tenant_id` column to all tables (or use `user_id` as tenant)
2. Enable RLS on all tables in Postgres: `ALTER TABLE conversations ENABLE ROW LEVEL SECURITY;`
3. Create policies: `CREATE POLICY tenant_isolation ON conversations USING (user_id = current_setting('app.user_id')::text);`
4. In the Next.js API, set `app.user_id` before each query: `await prisma.$executeRaw\`SET app.user_id = ${userId}\``
5. SQLite path: add `user_id` filter to all queries (no RLS in SQLite)
6. Test: user A cannot read user B's conversations

**Risks:**
- RLS policies are easy to get wrong (a missing policy = no access, not access to all)
- Performance: RLS adds a filter to every query
- SQLite fallback needs separate logic (application-level filtering)

**Validation criteria:**
- User A cannot access User B's data via any API route
- Performance impact < 5% on typical queries
- SQLite fallback enforces isolation at the application level

### Bet 5: Add real-time collaborative research with WebSocket

**Problem:** Research is currently a single-user activity. Multiple users can't collaborate on a research task in real-time.

**State of the art:**
- **Yjs** (https://github.com/yjs/yjs) — CRDT library for real-time collaborative editing. Used by Notion, Evernote, and Atlas. 18K stars. Supports text, JSON, and arbitrary data structures.
- **Liveblocks** (https://liveblocks.io/) — Real-time collaboration SaaS built on top of Yjs. Handles presence, comments, and document sync.
- **PartyKit** (https://partykit.io/) — Edge-rendered real-time multiplayer using Durable Objects. Open source. Simpler than Yjs for presence-only features.

**Implementation approach:**
1. Add a Socket.io mini-service (already supported by the project's gateway architecture)
2. When a user starts a research job, create a "room" with the job ID
3. Other users can join the room and see the research progress in real-time
4. Use Yjs for collaborative editing of the final report (multiple users can edit sections)
5. Add presence indicators (who's viewing, who's editing)

**Risks:**
- WebSocket connections are expensive to maintain (memory per connection)
- Conflict resolution in collaborative editing is complex (Yjs handles this but adds ~50KB to bundle)
- The research pipeline is currently synchronous (one job, one output) — making it collaborative requires rethinking the ownership model

**Validation criteria:**
- 5 users can view a research job simultaneously with < 500ms latency
- 2 users can edit the report simultaneously without conflicts
- WebSocket disconnection doesn't lose data (Yjs CRDT guarantees)

## Deep research notes

### Bet 1: WebAssembly sandbox (Pyodide + QuickJS-WASM)

The Node.js `vm` module is explicitly documented as unsafe: "The node:vm module is not a security mechanism. Do not use it to run untrusted code" (https://nodejs.org/api/vm.html). The current `code-sandbox.ts` uses `vm.createContext()` with a restricted sandbox object, but prototype chain escapes are well-documented (https://github.com/patriksimek/vm2 is abandoned precisely because escapes kept being found).

Pyodide (https://pyodide.org/) solves this for Python by compiling CPython to WebAssembly. The WASM sandbox has no filesystem access, no network access, and no `os` module by default. Pyodide is used in production by JupyterLite (https://jupyterlite.readthedocs.io/), CoCalc (https://cocalc.com/), and PythonAnywhere's in-browser execution. The main limitation is package availability: only pure-Python packages work without recompilation. NumPy, pandas, and sciPy are available (pre-compiled to WASM), but packages with C extensions may not work.

For JavaScript, QuickJS (https://bellard.org/quickjs/) compiled to WebAssembly provides a true sandbox. The wasmer-js runtime (https://github.com/wasmerio/wasmer-js) can execute WASI modules in Node.js with filesystem isolation. An alternative is Shiika (https://github.com/shiika-lang/shiika) or wazero (https://github.com/tetratelabs/wazero) for Go-based WASM runtimes, but for a Node.js project, wasmer-js is the natural choice.

The implementation cost is moderate: Pyodide loads in ~2s (one-time, cached), and execution is ~1.5x native speed. For the 10-second timeout use case in the code sandbox, this is well within bounds. The WASM modules are ~10MB (Pyodide) and ~2MB (QuickJS-WASM), which is smaller than the Docker daemon requirement.

### Bet 2: pgvector semantic search

The current LIKE search is a known limitation documented in README. pgvector (https://github.com/pgvector/pgvector) is the de facto standard for vector search in Postgres, with 9.5K GitHub stars and support from all major Postgres-as-a-service providers (Supabase, Neon, Railway, Render). It supports cosine distance (`<=>`), L2 distance (`<->`), and inner product (`<#>`), with HNSW (Hierarchical Navigable Small World) indexes for approximate search at scale.

Supabase's documentation on pgvector (https://supabase.com/docs/guides/ai/vector-columns) provides a practical guide for storing embeddings and performing similarity search. The typical pattern: store embeddings as `vector(1536)` (OpenAI) or `vector(1024)` (NVIDIA nv-embed-v1), create an HNSW index with `m=16, ef_construction=64`, and query with `ORDER BY embedding <=> query_embedding LIMIT 10`.

For hybrid search (combining vector similarity with keyword matching), Postgres can do this natively: `SELECT * FROM memories ORDER BY embedding <=> $1 * 0.7 + ts_rank(content, $2) * 0.3 DESC LIMIT 10`. This combines semantic and lexical search without a separate search engine. The `pg_trgm` extension (built into Postgres) provides trigram-based fuzzy matching as an alternative to full-text search.

For embedding generation, NVIDIA nv-embed-v1 (free via NIM API) produces 1024-dimensional embeddings. OpenAI text-embedding-3-small produces 1536-dimensional embeddings at $0.02/1M tokens. For a memory system with ~1000 memories, the embedding cost is negligible (<$0.01).

### Bet 3: Streaming collaborative agents

Multi-agent frameworks have matured significantly. Microsoft AutoGen (https://github.com/microsoft/autogen, 32K stars) demonstrates that agents can have structured conversations where one agent's output becomes another's input. The key insight from AutoGen's design is that agents should be able to "talk" to each other, not just produce independent outputs that a synthesizer combines.

CrewAI (https://github.com/joaomdmoura/crewai, 19K stars) takes a different approach: a "crew" of agents works through tasks sequentially or hierarchically, with a manager agent delegating and reviewing. The hierarchical model is closer to what Quaesitor needs: a lead agent that can ask workers to revise their outputs.

The current `SwarmCard.tsx` already has the UI for per-worker panels — the streaming infrastructure exists in `swarm.ts` (SSE events: `agent_token`, `agent_tool`, `agent_result`). What's missing is:
1. A "review" phase where the lead agent reads all worker outputs and asks specific workers for clarification
2. A "critic" agent that reviews the synthesized report before delivery

The cost concern is real: adding a review phase could 2-3x the LLM calls. However, the quality improvement from multi-round agent collaboration is well-documented in the ChatDev paper (https://arxiv.org/abs/2307.07924) — multi-agent collaboration produces higher-quality output than single-agent generation.

### Bet 4: Multi-tenant RLS

Postgres Row-Level Security is the gold standard for multi-tenant isolation. Supabase's multi-tenancy guide (https://supabase.com/docs/guides/auth/row-level-security) documents the pattern thoroughly: enable RLS, create policies that filter by `auth.uid()`, and the database enforces isolation even if the application has a bug.

The key advantage of RLS over application-level filtering is that it's defense-in-depth: even if a developer forgets to add a `WHERE user_id = ?` clause, the database still prevents cross-tenant access. The performance impact is minimal (<5% for typical queries) because Postgres optimizes RLS policies into the query plan.

For SQLite, there's no RLS equivalent. The application must enforce isolation via `WHERE` clauses. This is acceptable for development/single-user but should be documented as a limitation.

The Prisma multi-tenancy guide (https://www.prisma.io/docs/guides/tenant-multi-tenancy) shows how to use Prisma Client Extensions for tenant filtering. This is less secure than RLS (application-level) but works across both Postgres and SQLite.

### Bet 5: Real-time collaboration (Yjs + WebSocket)

Yjs (https://github.com/yjs/yjs, 18K stars) is the most mature CRDT library for JavaScript. It's used by Notion, Evernote, Atlas, and many collaborative editors. The key feature: CRDTs (Conflict-free Replicated Data Types) guarantee that concurrent edits converge to the same state without a central server resolving conflicts.

The Yjs documentation (https://docs.yjs.dev/) provides a getting-started guide for integrating with React. The typical pattern: create a `Y.Doc`, bind it to a text editor (e.g., TipTap, CodeMirror), and sync via a WebSocket provider. The y-websocket package (https://github.com/yjs/y-websocket) provides a simple WebSocket server.

For Quaesitor, the collaborative editing use case is: multiple users edit the research report simultaneously. Each section (from the plan) is a separate Yjs text fragment. When one user edits a section, the changes propagate to all connected users in real-time.

The PartyKit alternative (https://partykit.io/) is simpler for presence-only features (showing who's online) but doesn't handle collaborative editing. For full collaboration, Yjs is the right choice.

The main risk is bundle size: Yjs adds ~50KB (minified) to the client bundle. For a research tool, this is acceptable. The WebSocket server adds memory overhead (~1MB per 100 concurrent connections), but the project already has a gateway architecture that supports WebSocket via Socket.io.
