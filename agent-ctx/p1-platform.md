## What was implemented

### FEATURE 1 ‚Äî API Keys + `requireApiKey` (Developer Platform)

**Files changed:** `src/lib/auth.ts`, `src/lib/audit.ts`, `src/lib/db.ts`, `src/app/api/keys/route.ts` (new), `src/app/api/keys/[id]/route.ts` (new), `src/app/api/v1/chat/route.ts` (new).

- **`api_keys` table** added to `initSqliteSchema` in `src/lib/db.ts` (eager creation) AND lazily via `ensureApiKeysTable()` in `auth.ts` (defense-in-depth ‚Äî fresh DBs that skipped the schema init still work). Columns: `id`, `user_id`, `key_hash` (UNIQUE), `key_prefix`, `name`, `last_used_at`, `created_at`, `expires_at`. The raw key is NEVER stored ‚Äî only its SHA-256 hash. Indexes on `user_id` and `key_hash` for O(log n) lookup.
- **`requireApiKey(req)`** in `src/lib/auth.ts` ‚Äî validates a `Bearer qaesitor_...` Authorization header against the `api_keys` table. Returns `{ userId }` on success or a 401/500 `NextResponse` on failure. On success, stamps `last_used_at` (best-effort ‚Äî failure to update the timestamp does NOT block the request). Lookup is by SHA-256 hash, never by raw key.
- **`GET /api/keys`** ‚Äî lists the caller's API keys (masked). Response shape: `{ keys: [{ id, name, keyPrefix, lastUsedAt, createdAt, expiresAt }], total }`. Auth: `requireAuth` + `getUserId` (Basic auth, NOT API-key auth ‚Äî these routes are part of the dashboard UI, not the public API).
- **`POST /api/keys`** ‚Äî creates a new API key. Body: `{ name: string }`. Key format: `qaesitor_${randomBytes(24).toString("base64url")}` (~32 chars of entropy after the prefix). Returns the raw key EXACTLY ONCE in the response (`{ ok, key: { id, name, key, keyPrefix, createdAt, lastUsedAt, expiresAt }, warning }`) along with a "store it securely ‚Äî it cannot be recovered" warning. The raw key is never logged. The `key_prefix` (e.g. `qaesitor_abc1‚Ä¢‚Ä¢‚Ä¢‚Ä¢`) is stored at creation time so the listing UI can identify the key without exposing the secret.
- **`DELETE /api/keys/[id]`** ‚Äî revokes (hard-DELETE) an API key. Verifies ownership BEFORE deleting (404, not 403, when the key doesn't exist or belongs to another user ‚Äî avoids leaking other users' key IDs). Returns `{ ok, id }` on success.
- **Audit logging**: added `"apikey.create"` and `"apikey.delete"` to `SENSITIVE_ACTIONS` in `src/lib/audit.ts` (resource: `"apikey"`). Both routes call `logSensitiveAction(...)` with the `keyId`, `keyPrefix`, and `name` in metadata ‚Äî enough to reconstruct which key was rotated without exposing the secret.
- **`POST /api/v1/chat`** ‚Äî public API endpoint using `requireApiKey` instead of `requireAuth`. Same SSE token stream shape as `/api/chat` (so SDKs that already parse that stream work unchanged). Records usage via `recordUsage(userId, "chat")` for metered billing. Includes the same prompt-injection defense and plan-limit enforcement (402 on limit reached) as the dashboard route. Memory recall is intentionally NOT wired up ‚Äî the v1 API is a low-level primitive for integrators; silently injecting recalled memories into a programmatic request would be surprising.

### FEATURE 2 ‚Äî NLI Citation Verifier via LLM

**Files changed:** `src/lib/citation-verifier.ts`.

The existing `verifyCitation` (sync) and `verifyAllCitations` (sync) are UNCHANGED ‚Äî the test suite (19 tests) and existing callers (research-engine, eval runner) continue to use them with no behavioral change.

Added an LLM-backed NLI layer accessed via two NEW async wrappers:

- **`verifyCitationWithNLI(url, citedText, sources): Promise<CitationCheck>`** ‚Äî runs the sync `verifyCitation`, then if NLI is enabled and the source has text content, calls `verifyWithNLI(citedText, sourceText)` and attaches the verdict as `nliVerdict` on the returned `CitationCheck` (alongside, not replacing, the sync `supportsClaim`).
- **`verifyAllCitationsWithNLI(report, sources): Promise<VerificationReport>`** ‚Äî same, for all citations in a report. Runs NLI for each citation concurrently via `Promise.all`.
- **`verifyWithNLI(claim, sourceText)`** (internal) ‚Äî calls `llm.fast()` with the `NLI_PROMPT` system prompt and a `CLAIM: ... SOURCE TEXT: ... Verdict:` user message. Claim is truncated to 500 chars, source to 2000 chars (bounded token cost). Parses the model's response by keyword inclusion ("support" ‚Üí supports, "contradict" ‚Üí contradicts, else irrelevant). NEVER throws ‚Äî on any failure, returns "irrelevant" (fail safe: a broken LLM never upgrades a citation to "supports"). The `llm-provider` module is dynamically imported (`await import("./llm-provider")`) so the test suite (which imports this file for the sync `verifyCitation`) doesn't pull the LLM provider graph into the test bundle.
- **`CitationCheck.nliVerdict?: "supports" | "contradicts" | "irrelevant"`** ‚Äî new optional field. Populated only when NLI is enabled AND the async wrappers are used. The sync `verifyCitation` does NOT populate it.
- **Gating**: `isNliVerifierEnabled()` returns `process.env.NLI_VERIFIER_ENABLED === "true"`. OFF by default ‚Äî the test suite (which exercises the sync `verifyCitation`) never makes real LLM calls. When disabled, the async wrappers are equivalent to their sync counterparts (just wrapped in a Promise).
- **Caching**: in-memory `Map<string, NliCacheEntry>` keyed by `nli:${sha256(claim+source).slice(0,32)}`. TTL: 7 days. Bounded to 10_000 entries with coarse LRU-ish eviction (drops oldest 25% when full). Lazy TTL eviction on read. `clearNliCache()` exported for tests. Cache lookup happens BEFORE the dynamic import ‚Äî a cache hit avoids both the LLM call AND the import cost.

### FEATURE 3 ‚Äî Stripe Metered Billing

**Files changed:** `src/lib/stripe.ts`, `src/lib/usage-tracker.ts` (new), `src/app/api/billing/usage/route.ts`.

- **`reportUsage(subscriptionItemId, quantity, customerId?)`** in `src/lib/stripe.ts` ‚Äî reports metered usage to Stripe. **API version compatibility**: the installed stripe-node v22 removed `subscriptionItems.createUsageRecord` from the TypeScript types (deprecated in v17 in favor of the new `billing.meterEvents.create` event-name-based API). The function probes for the legacy method at runtime via a typed cast (`as unknown as { createUsageRecord?: ... }`) ‚Äî if present (older stripe-node installs or operators who pinned the legacy API), it uses the spec's original API. Otherwise it falls back to the new `billing.meterEvents.create` API, mapping the meter event name via the `STRIPE_METER_EVENT_NAME` env var (default: `quaesitor_usage`) and the customer id via the `customerId` parameter. The third `customerId` parameter is optional (backward compat with the spec's 2-arg signature) ‚Äî when omitted and the legacy API is unavailable, the report is dropped with a warn-level log. Failures are swallowed (logged at error level) ‚Äî usage reporting is best-effort and must NOT block the request.
- **`src/lib/usage-tracker.ts`** (new) ‚Äî exposes `recordUsage(userId, type)` (synchronous, persists to the existing `usage_records` table immediately so usage data survives process restarts), `flushUsage()` (async, reads all unreported usage from the DB, groups by subscription, reports to Stripe), `startUsageFlusher()` / `stopUsageFlusher()` (lifecycle). Auto-starts a 60-second interval on module import ‚Äî but ONLY when `STRIPE_SECRET_KEY` is set (dev/test deployments without Stripe never start the timer, so the test suite is unaffected). The timer is `unref()`'d so it never keeps the process alive on its own (test runner can exit cleanly between files). The flusher is a simplified version: it walks every active subscription with a `stripe_subscription_item_id` column, sums the user's chat + research + swarm counts for the current period, and reports the aggregate as a single `reportUsage` call. When the `stripe_subscription_item_id` column doesn't exist yet (migration not applied), `flushUsage` is a no-op ‚Äî the DB-side usage tracking still works independently, so plan-limits enforcement and the `/api/billing/usage` endpoint continue to function.
- **`GET /api/billing/usage`** ‚Äî updated to include the new P1 fields `currentPeriod` (chat/research/swarm counts), `projectedCost` (linear extrapolation based on day-of-month), and `limits` (per-type limits for the user's plan from `PLAN_LIMITS`). The legacy fields (`plan`, `planName`, `usage`, `features`) are kept for backward compatibility ‚Äî existing dashboard code that reads them continues to work. The projected cost is currently the plan's flat monthly price (overage pricing per-unit-over-quota is configured in Stripe, not in the app ‚Äî when Stripe-side overage pricing is wired up, the projection can be extended). Free plan returns `$0.00`. Unlimited plans (team/enterprise) return the flat price.

## Notes for downstream agents

- **API key auth is a separate auth surface from Basic auth.** The `/api/v1/*` namespace uses `requireApiKey` exclusively; the `/api/keys/*` dashboard routes use `requireAuth` + `getUserId` (you need Basic auth to create an API key ‚Äî chicken-and-egg otherwise). New public-API routes should follow the `/api/v1/*` pattern: `const apiAuth = requireApiKey(req); if (apiAuth instanceof Response) return apiAuth; const userId = apiAuth.userId;`. The `instanceof Response` check narrows the `{ userId } | NextResponse` union correctly because `NextResponse extends Response`.
- **The `api_keys` table is created eagerly AND lazily.** Eager: `initSqliteSchema` in `src/lib/db.ts` (so the table exists on first startup of a fresh DB). Lazy: `ensureApiKeysTable()` in `src/lib/auth.ts` and in both `/api/keys` route handlers (defense-in-depth ‚Äî a Postgres-mode deployment that skipped the SQLite schema init still works because the routes create the table on first call). When migrating to Postgres-only, add a Prisma `ApiKey` model and use `prisma.apiKey.*` instead of `getDb()`.
- **`recordUsage(userId, type)` is the canonical metered-billing entry point.** Every new route that performs a metered action (chat, research, swarm, future: vision, file-generation) should call it. The 60-second flusher handles Stripe reporting ‚Äî routes should NOT call `reportUsage` directly. `recordUsage` is synchronous and best-effort (failures are swallowed) so it never blocks the user-facing request.
- **NLI is opt-in via `NLI_VERIFIER_ENABLED=true`.** The async wrappers `verifyCitationWithNLI` / `verifyAllCitationsWithNLI` are the production-grade entry points (deeper paraphrase/antonym understanding via the LLM). The sync `verifyCitation` / `verifyAllCitations` remain the fast path and are the only ones the test suite exercises. Wire the async wrappers into `research-engine` / `deep-research` flows when NLI is enabled ‚Äî they're already additive (the `nliVerdict` field sits alongside `supportsClaim`, doesn't replace it).
- **`reportUsage` has a 3rd optional `customerId` parameter** for the new Stripe meter-events API (v17+). The legacy 2-arg signature still works when the runtime has `subscriptionItems.createUsageRecord` (older stripe-node or pinned legacy API). When calling `reportUsage` from a context that has the customer id available, pass it ‚Äî it enables the new-API fallback path.
- **The `subscriptions` table needs a `stripe_subscription_item_id` column** before `flushUsage()` actually reports to Stripe. That column would be populated by the `checkout.session.completed` webhook handler (currently it stores `stripe_subscription_id` but not the subscription item id). Adding the column + populating it in the webhook is the natural next step to fully activate metered billing ‚Äî the flusher is already written to detect the column's absence and no-op gracefully.
- **The `‚Ä¢‚Ä¢‚Ä¢‚Ä¢` bullet character is fine in code strings but NOT inside backtick template literals that contain SQL** ‚Äî the template literal would close at any internal backtick. The `api_keys` schema comment in `db.ts` originally had backticks around `key_prefix` which closed the template literal mid-SQL and broke `tsc`. Fixed by removing the backticks from the comment. Future agents adding SQL comments inside template literals: avoid backticks, or use string concatenation.
